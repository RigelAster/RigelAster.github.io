Assets, Resources and AssetBundles - Ch.3 The Resources folder

发表于2025-06-09|更新于2025-06-09|Unity
|浏览量:

原文链接：https://learn.unity.com/tutorial/assets-resources-and-assetbundles

读前须知：这是一篇在Unity 2017.3语境下所写的文章，目前已被标记为deprecated。官方更推荐使用新推出的Addressables系统，但是本文所蕴含的知识应该不会过时吧，一下为原文提示：

PLEASE NOTE: this tutorial has now been deprecated. We now recommend using Addressables for your projects and will be providing more documentation and tutorials on this feature soon. In the meantime please refer to the Unity manual

提示：本教程中提到的Objects与Assets与Unity所提供API使用的命名约定不同。在本教程中被称为Objects的数据，在Unity APIs中被称为Assets，例如 AssetBundle.LoadAsset和Resources.UnloadUnusedAssets。在本教程中被称为Assets的文件，则很少会被暴露在公共APIs中。

Note: This guide's terms for Objects and Assets differ from Unity's public API naming conventions.

The data this guide calls Objects are called Assets in many public Unity APIs, such asAssetBundle.LoadAsset andResources.UnloadUnusedAssets. The files this guide calls Assets are rarely exposed to any public APIs. When they are exposed, it is generally only in build-related code, such asAssetDatabase andBuildPipeline. In these cases, they are called files in public APIs.

The Resources folder

本章节讨论Resources系统，它允许开发人员将资产存储在一个或多个名为Resources的文件夹中，并在运行时使用Resource API从这些资产中加载或卸载对象。

Best Practices for the Resources System

不要使用它

Don't use it.

提出这一过激建议有以下几个原因：

  • 使用Resouces文件夹会使细粒度内存管理变得更加困难
  • 不正确地使用资源文件夹会增加应用程序的启动时间和构建长度
    • 随着Resource文件夹数量的增加，管理这些文件夹中的资产变得非常困难
  • Resources系统降低了项目向特定平台交付定制内容的能力，并消除了增量内容升级的可能性
    • AssetBundle Variants是Unity在每个设备上调整内容的主要工具

Proper Uses of the Resources system

Resources系统在以下两种情况下，可以在不妨碍良好开发实践的情况下提供帮助：

  1. Resoures文件夹的易用性使其成为快速创建原型的绝佳系统。但是，当项目进入完整的生产环境时，应该避免对Resources文件夹的使用。
  2. Resources文件夹在一些不重要的情况下可能很有用：
  3. 在项目的整个生命周期中都需要（Generally required throughout a project's lifetime）
  4. 不是内存密集型（Not memory-intensive）
  5. 不容易打补丁，或者不会因平台或设备而变化（Not prone to patching, or does not vary across platforms or devices）
  6. 用于最小引用（Used for minimal bootstrapping）

第二种情况的例子包括用于托管预制体的Mono单例，或者包含第三方配置数据的ScriptableObjects，比如Facebook App ID。

Serialization of Resources

当构建项目时，所有名为Resources文件夹中的资产和对象被组合成单个序列化文件。该文件还包含metadata和索引信息，类似于AssetBundle。如AssetBundle文档中所述，该索引包括一个序列化的查找树，用于将给定对象的名称解析为相应的File GUID和Local ID。此外，它还用于在序列化文件体中的特定字节位移位置定位Object。

在大多数平台上，查找数据结构是一个平衡树(a balanced search tree)，它的构建时间以O(nlogn)的速度增长。这种增长还会导致索引的加载时间随着Resources文件夹中对象数量的增加而非线性(more-than-linearly)增长。

此操作不可跳过，并且发生在应用程序启动时，当显示初始非交互式启动面板时。在低端移动设备上，初始化包含10,000个资源的Resouces系统需要花费数秒，尽管Resources文件夹中包含的大多数对象实际上很少需要加载到应用程序的第一个场景中。
