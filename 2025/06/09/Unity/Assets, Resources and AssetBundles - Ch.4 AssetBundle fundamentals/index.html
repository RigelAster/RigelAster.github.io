Assets, Resources and AssetBundles - Ch.4 AssetBundle fundamentals

发表于2025-06-09|更新于2025-06-09|Unity
|浏览量:

原文链接：https://learn.unity.com/tutorial/assets-resources-and-assetbundles

读前须知：这是一篇在Unity 2017.3语境下所写的文章，目前已被标记为deprecated。官方更推荐使用新推出的Addressables系统，但是本文所蕴含的知识应该不会过时吧，一下为原文提示：

PLEASE NOTE: this tutorial has now been deprecated. We now recommend using Addressables for your projects and will be providing more documentation and tutorials on this feature soon. In the meantime please refer to the Unity manual

提示：本教程中提到的Objects与Assets与Unity所提供API使用的命名约定不同。在本教程中被称为Objects的数据，在Unity APIs中被称为Assets，例如 AssetBundle.LoadAsset和Resources.UnloadUnusedAssets。在本教程中被称为Assets的文件，则很少会被暴露在公共APIs中。

Note: This guide's terms for Objects and Assets differ from Unity's public API naming conventions.

The data this guide calls Objects are called Assets in many public Unity APIs, such asAssetBundle.LoadAsset andResources.UnloadUnusedAssets. The files this guide calls Assets are rarely exposed to any public APIs. When they are exposed, it is generally only in build-related code, such asAssetDatabase andBuildPipeline. In these cases, they are called files in public APIs.

AssetBundle fundamentals

本章讨论AssetBundles，介绍了构建AssetBundles的基本系统，以及用于与AssetBundle交互的核心API。特别地，还会讨论AssetBundles本身的加载、卸载，以及从AssetBundle中加载和卸载特定的Asset和Objects。

Overview

AssetBundle系统提供了一种以存档格式存储一个或多个文件的方法，Unity可以对其进行索引和序列化。AssetBundles是Unity的主要工具，用于在安装后交付和更新非代码内容。这允许开发者提交更小的应用程序包，最小化运行时内存压力，并选择性地加载针对终端用户设备优化的内容。

理解AssetBundles的工作方式对于为移动设备构建一个成功的Unity项目至关重要。有关AssetBundle内容的总体描述，请查看AssetBundle文档。

AssetBundle layout

总而言之，一个AssetBundle由两部分组成：头部和数据段（a header and data segment）。

头部(header)包含有关AssetBundle的信息，例如它的标识符、压缩类型和清单。清单(manifest)是一个以对象名称为键的查找表。每个条目提供一个字节索引(a byte index)，指示在AssetBundle的数据段中可以找到给定对象的位置。在大多数平台上，这个查找表是作为平衡搜索树实现的。具体来说，Windows和OSX派生平台（包括iOS）使用红黑树。因此，构建清单所需的时间将随着AssetBundle中资产数量的增加而非线性(more than linearly)增加。

数据段(data segment)包含通过在AssetBundle中序列化资产而生成的原始数据。如果将LZMA指定为压缩方案，则压缩所有序列化资产的完整字节数组。如果指定LZ4，则单独压缩不同资产的字节。如果不使用压缩，数据段将保留为原始字节流。

在Unity 5.3之前，Objects不能在AssetBundle中单独压缩。因此，如果5.3之前的Unity版本被指示从压缩的AssetBundle中读取一个或多个Objects，Unity必须解压缩整个AssetBundle。通常，Unity会缓存AssetBundle的解压缩副本，以提高对同一AssetBundle的后续加载请求的加载性能。

Loading AssetBundles

AssetBundles可以通过四个不同的API加载，这四个API的行为取决于两个标准：

  1. AssetBundle的压缩类型，是LZMA、LZ4或者uncompressed
  2. AssetBundle被加载的平台platform

这四个API分别是：

  • AssetBundle.LoadFromMemory(Async optional)
  • AssetBundle.LoadFromFile(Async optional)
  • UnityWebRequest's DownloadHandlerAssetBundle
  • WWW.LoadFromCacheOrDownload(on Unity 5.6 or older)

AssetBundle.LoadFromMemory(Async)

Unity的建议是不要使用此API

Unity's recommendation is not to use this API.

AssetBundle.LoadFromMemroyAsync通过托管代码字节数组(a managed-code byte array, byte[] in C#)来加载AssetBundle。他总是将源数据从托管代码字节数组复制到新分配的、连续的本机内存块中。如果AssetBundle是LZMA compressed的，会在复制时解压缩。UnCompressed 和 LZ4-compressed 的将被逐字复制。

此API消耗的内存峰值至少是AssetBundle大小的两倍：一个拷贝在API创建的本机内存中，另一个拷贝在传递给API的托管字节数组中。因此，通过此API创建的AssetBundle加载的资产将在内存中复制三次：一次在托管代码字节数组中，一次在AssetBundle的本机内存副本中，第三次在GPU或系统内存中为资产本身的复制。

Prior to Unity 5.3.3, this API was known as AssetBundle.CreateFromMemory. Its functionality has not changed.

AssetBundle.LoadFromFile(Async)

AssetBundle.LoadFromFile是一个高效的API，用于从本地存储（如硬盘或SD卡）加载uncompressed 或 LZ4-compressed AssetBundle。

在 desktop standalone, console, mobile 上，此API将只加载AssetBundle的头部，而将剩余的数据留在磁盘上。当加载方法（例如AssetBundle.Load）被调用或它们的InstanceIDs被解引用时，AssetBundle的Objects将按需加载。此场景中，不会消耗额外的内存。在Unity Editor中，API会将整个AssetBundle加载到内存中，就像从磁盘读取字节一样，AssetBundle.LoadFromMemoryAsync将被调用。如果项目是在Unity Editor调试的，此API可能会导致在加载AssetBundle期间出现内存峰值。这应该不会影响在设备上的性能，在采取补救措施之前，应该在设备上重新测试这些峰值。

Note: 在使用Unity 5.3或更早版本的Android设备上，当尝试从Streaming Assets路径加载AssetBundles时，此API将失败。这个问题已经在Unity 5.4中解决了。要了解更多细节，请参阅"Distribution - shipped with project"”一节。

Prior to Unity 5.3, this API was known as AssetBundle.CreateFromFile. Its functionality has not been changed.

AssetBundleDownloadHandler

UnityWebRequest API允许开发人员指定Unity应该如何处理下载的数据，并允许开发人员消除不必要的内存使用。使用UnityWebRequest下载AssetBundle最简单的方法是调用函数UnityWebRequest.GetAssetBundle。

本文中，我们专注于DownloadHandlerAssetBundle类。使用worker thread，它将下载的数据流式传输到固定大小的缓冲区中，然后根据下载处理程序的配置方式，将缓冲的数据转到临时存储或AssetBundle缓冲中。所有这些操作都在本机代码中进行，从而消除了扩展托管堆的风险。此外，这个下载处理程序不保留所有下载字节的本地代码副本，进一步减少下载AssetBundle的内存开销。

LZMA-compressed AssetBundles 将在下载过程中解压缩，并使用 LZ4 compression 进行缓存。此行为可以通过设置 Caching.CompressionEnabled 来调整。

当下载完成后，Download Handler 的 assetBundle 属性提供了对下载的AssetBundle的访问，就像AssetBundle.LoadFromFile在下载的AssetBundle上被调用。

如果缓存信息提供给一个UnityWebRequest对象，并且请求的AssetBundle已经存在于Unity的缓存中，那么该AssetBundle将立即可用，这个API作用将于AssetBundle.LoadFromFile相同。

在Unity5.6之前，UnityWebRequest系统使用一个固定的worker thread池和内部作业系统来防止过多的并发下载。thread池的大小是不可配置的。在Unity5.6中，这些保护措施已被移除，以适应更现代的硬件，并允许更快地访问HTTP响应代码和头部。

WWW.LoadFromCacheOrDownload

Note: Beginning in Unity 2017.1, WWW.LoadFromCacheOrDownload simply wraps around UnityWebRequest. Accordingly, developers using Unity 2017.1 or higher should migrate to UnityWebRequest. WWW.LoadFromCacheOrDownload will be deprecated in a future release.

The following information is applicable to Unity 5.6 or older.

WWW.LoadFromCacheOrDownload 是一个允许从远程服务器和本地存储加载对象的API。文件可以通过 file:// URL 从本地存储加载。如果AssetBundle存在于Unity缓存中，这个API的行为和 AssetBundle.LoadFromFile 相同。

如果一个AssetBundle还没有被缓存，那么WWW.LoadFromCacheOrDownload将从源文件中读取AssetBundle。如果AssetBundle被压缩，它将使用worker thread解压缩并写入缓存。否则，它将通过worker thread直接写入缓存。一旦AssetBundle被缓存，WWW.LoadFromCacheOrDownload将从缓存、解压缩的AssetBundle中加载头部信息。然后此API的行为将与被AssetBundle. loadfromfile加载的AssetBundle相同。此缓存在WWW.LoadFromCacheOrDownload 和UnityWebRequest之间共享。用一个API下载的任何AssetBundle也可以通过另一个API使用。

当数据将被解压缩并通过固定大小的缓冲区写入缓存时，WWW对象将在本机内存中保留AssetBundle字节的完整副本。保存这个额外的AssetBundle副本是为了支持 WWW.bytes 属性。

由于在WWW对象中缓存AssetBundle字节的内存开销，AssetBundle应该保持小——最多几兆字节。有关AssetBundle大小的更多讨论，请参阅AssetBundle usage patterns章节中的Asset assignment strategies 部分。

不像UnityWebRequest，每次调用这个API都会产生一个新的worker thread。因此，在内存有限的平台上，比如移动设备，一次只应该使用这个API下载一个AssetBundle，以避免内存峰值。在多次调用此API时，要小心创建过多的线程。如果需要下载超过5个AssetBundle，在脚本代码中创建并管理一个下载队列，以确保只有几个AssetBundle下载同时运行。

Recommendations

一般来说，应尽可能使用AssetBundle.LoadFromFile。这个API在速度、磁盘使用和运行时内存使用方面是最高效的。

对于必须下载或修补AssetBundles的项目，强烈建议在Unity 5.3或更新版本的项目中使用UnityWebRequest，在Unity 5.2或更老版本使用 WWW.LoadFromCacheOrDownload。正如Distribution部分所详述的那样，可以用项目安装程序中包含的Bundle来准备AssetBundle Cache。

当使用 UnityWebRequest 或 WWW.LoadFromCacheOrDownload 时，确保下载器代码在加载AssetBundle后正确调用 Dispose 函数。C#的using语句是确保WWW或UnityWebRequest被安全卸载的最方便的方法。

Loading Assets From AssetBundle

UnityEngine.Objects 可以使用三个不同的API从AssetBundle加载，它们都有同步和异步的变体：

  • LoadAsset LoadAssetAsync
  • LoadAllAssets LoadAllAssetsAsync
  • LoadAssetWithSubAssets LoadAssetWithSubAssetsAsync

这些API的同步版本总是比它们的异步版本快，至少快一帧。

异步加载将每帧加载多个对象，受到时间片限制。有关更底层技术原因，参考 Low-level loading details 一节。

LoadAllAssets 应该在加载多个独立 UnityEngine.Objects 时使用。它只应该在需要加载AssetBundle中的大部分或所有对象时使用。相比于其他两个API，LoadAllAssets比多次单独调用 LoadAssets要快一些。因此，如果需要加载的资产数量很大，但一次需要加载的资产少于66%，请考虑将AssetBundle拆分为多个较小的bundles，并使用 LoadAllAssets。

LoadAssetWithSubAssets应该在加载包含多个嵌入对象的复合资产时使用，例如带有嵌入动画的FBX模型或带有多个嵌入精灵的精灵图集。如果需要加载的所有对象都来自同一个资产，但是与许多其他不相关的对象一起存储在AssetBundle中，那么使用这个API。

对于其他场景，使用 LoadAsset 和 LoadAssetAsync。

Low-level loading details

UnityEngine.Object 加载是在主线程之外执行的：对象数据通过工作线程(worker thread)从存储中读取。任何不涉及Unity系统线程敏感部分（脚本，图形）的内容将在工作线程上进行转换。例如，VBOs将从网格中创建，纹理将被解压，等等。

从Unity 5.3开始，对象加载已经并行化。多个对象在工作线程上被反序列化、处理和集成。当一个对象完成加载时，它的Awake回调将被调用，并且该对象将在下一帧期间对Unity引擎的其余部分可用。

同步方法 AssetBundle.Load 将暂停主线程，直到对象加载完成。它们还将对对象加载进行时间切片，以便对象集成占用的帧时间不会超过一定的毫秒数。 毫秒数由Application.backgroundLoadingPriority 属性设置：

  • ThreadPriority.High: Maximum 50 milliseconds per frame
  • ThreadPriority.Normal: Maximum 10 milliseconds per frame
  • ThreadPriority.BelowNormal: Maximum 4 milliseconds per frame
  • ThreadPriority.Low: Maximum 2 milliseconds per frame.

从Unity 5.2开始，多个对象被加载，直到达到对象加载的帧时间限制。假设所有其他因素都相同，资产加载API的异步变体将总是比同步版本花费更长的时间来完成，因为在发出异步调用和对象对引擎可用之间的最小一帧延迟。

AssetBundle dependencies

根据运行时环境，使用两个不同的API自动跟踪AssetBundles之间的依赖关系。在Unity Editor中，可以通过AssetDatabase API查询AssetBundle的依赖关系。AssetBundle的分配和依赖可以通过AssetImporter API来访问和修改。运行时，Unity提供了一个可选API，通过基于ScriptableObject的AssetBundleManifest API加载在AssetBundle构建过程中生成的依赖信息。

当一个或多个UnityEngine.Objects引用一个或多个其他AssetBundle的UnityEngine.Objects时，其父AssetBundle便依赖于其他AssetBundle。有关对象间引用的更多信息，请查阅Assets, Objects and Serialization章节的 Inter-Object references 部分。如Serialization and instances部分所述，AssetBundle作为源数据的源，由AssetBundle中包含的每个对象的File GUID和Local ID标识。

因为一个对象是在它的Instance ID第一次被解引用时被加载的，而Instance ID又是在它的AssetBundle被加载时被分配的，所以AssetBundle加载的顺序并不重要。相反，在加载对象本身之前加载包含对象依赖的所有AssetBundles是很重要的。Unity不会尝试在加载父AssetBundle时自动加载任何子AssetBundle。

Example：

假设A材质引用了B纹理。A材质被打包进AssetBundle 1，B纹理被打包进AssetBundle 2。在这个案例中，AssetBundle 2必须在加载A材质之前被加载。

这并不表示AssetBundle 2必须在AssetBundle 1之前被加载，或B纹理必须从AssetBundle 2中显式加载。在从AssetBundle 1中加载A材质之前，加载AssetBundle 2就足够了。

但是，Unity不会在加载Asset Bundle 1时自动加载AssetBundle 2。这一过程必须通过脚本代码自主实现。

关于依赖的更过内容，请查阅 manual page 。

AssetBundle manifests

当使用 BuildPipeline.BuildAssetBundles API 执行AssetBundle构建管线时，Unity会序列化一个包含每个AssetBundle的依赖信息的对象。这些数据存储在一个单独的AssetBundle中，它包含一个AssetBundleManifest 类型的Object。

这个资产将被存储在一个与AssetBundle被构建时的父目录名称相同的AssetBundle中。如果一个项目将AssetBundles构建到 (projectroot)/build/Client/ 文件夹中，保存maifest的AssetBundle将被保存为(projectroot)/build/Client/Client.manifest 。

包含manifest的AssetBundle可以向其他AB一样被加载、缓存和卸载。

AssetBundleManifest对象本身提供了GetAllAssetBundles API来列出所有与mainfest一同构建的AssetBundle，以及两个方法来查询特定AssetBundle的依赖关系：

  • AssetBundleManifest.GetAllDependencies returns all of an AssetBundle's hierarchical dependencies, which includes the dependencies of the AssetBundle's direct children, its children's children, etc.
  • AssetBundleManifest.GetDirectDependencies returns only an AssetBundle's direct children

注意，这两个API都会分配字符串数组。因此，它们应该只被谨慎使用，而不是在应用程序生命周期中对性能敏感的部分使用。

Recommendations

在许多情况下，最好在玩家进入应用程序的性能关键区域（如主要游戏关卡或世界）之前加载尽可能多的所需对象。这在移动平台上尤其重要，因为在移动平台上访问本地存储很慢，并且在游戏时间加载和卸载对象的内存波动可能会触发垃圾收集器。

对于那些必须在应用程序交互时加载和卸载对象的项目，请参阅 AssetBundle usage patterns 步骤中的 Managing loaded assets 一节，了解有关卸载Objects和AssetBundle的更多信息。
